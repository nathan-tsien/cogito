# ADR-0022: Plugin distribution (git fetch + lock file)

## Status

Proposed — placeholder (finalized in v0.3).

Captures the v0.3 tier of the **P4 tiered plugin distribution** plan
from the
[2026-05-22 roadmap rebalance spec](../superpowers/specs/2026-05-22-roadmap-rebalance-design.md)
(§2.4 + §3.3). v0.2 ships **local path only** (ADR-0021); v0.3 adds
**git fetch + lock file** (this ADR); marketplace / HTTP / signing
are v0.6+ (future ADR).

## Context

By v0.3, plugin manifest schema (ADR-0021) has been used in real
projects for at least one v0.2 cycle; the schema is stable enough to
attach a distribution layer on top.

Distribution requirements for v0.3:

- A plugin lives in a **Git repository**, pinned by **tag or commit
  SHA**. No floating refs (`main`, `HEAD`) allowed in production
  configuration — they break reproducibility.
- The set of resolved versions across all `[[plugins]]` entries is
  recorded in a **lock file** (`cogito.lock`) checked into the
  consumer's repo. Lock file is the source of truth at runtime.
- Sync is **explicit**: `cogito plugin sync` fetches/updates;
  `cogito chat` does **not** make network calls.
- Failed fetches are **non-fatal at startup** if a cached copy is
  present and matches the lock; **fatal** if no cache.

Out of scope for v0.3:
- HTTP marketplace indexes (deferred to v0.6+)
- Signing / signature verification
- Vendoring (consumer copies plugin source into repo — orthogonal,
  always allowed)
- Mirror / cache server protocols

## Decision

(Details locked at v0.3 sprint kickoff. Skeleton below.)

### 1. Plugin source syntax in `cogito.toml`

```toml
[[plugins]]
git = "https://github.com/org/cogito-plugin-review.git"
rev = "v0.1.0"            # tag OR full commit SHA; floating refs rejected

# Optional auth (mirrors MCP bearer pattern from ADR-0018):
ssh_key_env_var = "PLUGIN_DEPLOY_KEY"  # for SSH origins
token_env_var   = "GH_PAT"             # for HTTPS PATs
```

Either `path` (ADR-0021) or `git` is required; mutually exclusive.

### 2. Lock file: `cogito.lock`

```toml
# Auto-generated by `cogito plugin sync`. Do not edit manually.
schema_version = 1

[[plugin]]
git = "https://github.com/org/cogito-plugin-review.git"
declared_rev = "v0.1.0"
resolved_commit = "a1b2c3d4e5f6..."
resolved_at = "2026-XX-XXTXX:XX:XXZ"
content_hash = "sha256:..."
```

The lock file is **committed alongside `cogito.toml`** in the
consumer's repo. CI runs `cogito plugin sync --check` to verify the
lock is up to date.

### 3. Cache layout

```
~/.cache/cogito/plugins/
  <content-hash-prefix>/
    .cogito-plugin/plugin.toml
    skills/...
    ...
```

Keyed by `content_hash` (not by git URL or commit), so multiple
declarations of the same content share storage.

### 4. CLI commands

- `cogito plugin sync` — fetch any plugin in `cogito.toml` whose
  `resolved_commit` isn't in cache; update lock file
- `cogito plugin sync --check` — exit 0 if lock matches manifest,
  non-zero otherwise (CI use)
- `cogito plugin list` — display loaded plugins + resolved revisions
- `cogito plugin update <plugin_id>` — re-resolve one plugin's `rev`
  to its current ref (for tag movement / floating tag updates done
  by maintainer)

### 5. Runtime behavior

`cogito chat` / `cogito serve` / etc.:

- Read `cogito.toml` + `cogito.lock`
- For each `[[plugins]] git = "..."`: look up `resolved_commit` in
  cache → load from cache
- Cache miss + no network attempted at runtime → **fatal startup
  error** with instruction to run `cogito plugin sync`
- Lock missing entirely → **fatal startup error**

This guarantees `cogito chat` is **offline-capable** as long as the
cache is populated.

### 6. Network failure semantics during `sync`

- Single plugin fetch failure → recorded in `mcp_startup_failures`-style
  channel; sync continues for other plugins; exit code reflects partial
  failure
- All-fail → exit non-zero; user instructed to check network / auth

## Open questions (for v0.3 sprint)

- Submodule support: do we resolve git submodules during fetch, or
  treat plugin as a flat tree? (Recommendation: flat tree; submodules
  add CVE exposure.)
- LFS support: do we honor `.gitattributes` LFS pointers for binary
  resources (e.g., skill `assets/`)? (Likely yes; trivial via `git
  lfs fetch`.)
- Self-hosted git / non-GitHub origins: any special-casing or strict
  URL-only?
- Lock file conflict resolution (two devs edit `cogito.toml` then
  run sync): use cargo-style "preserve oldest resolution, update
  newest" or always re-resolve?

## Consequences

**Easier**:
- Plugins distributed via existing infrastructure (Git, no new
  hosting required)
- Reproducible builds: lock file pins exact source
- CI verification: `--check` mode is one-line gate
- Air-gapped deployments work: vendor caches via pre-populated
  `~/.cache/cogito/plugins/`

**Harder**:
- Adds Git as runtime dep (currently only build-time in cogito);
  Sprint may use `git2` Rust crate or shell out to `git`
- Lock file maintenance discipline (consumer responsibility)

**Given up**:
- Floating tags (`main`, `latest`) — explicit decision, follows
  cargo's lockfile model
- Cross-deployment plugin discovery without explicit URL — there's
  no `cogito plugin search <term>` until P3 marketplace
