# v0.1 Hook Pipeline Latency Baseline

> Measured against `cogito-core` H09 `CompositeHookPipeline` via
> `crates/cogito-core/benches/hook_latency.rs`.
>
> **SLO:** P99 < 5 ms per hook pipeline invocation (H09 §"Critical invariants").

## Methodology

- Benchmark tool: Criterion 0.5 (100 samples, 3 s warmup, ~5 s collection window)
- Pipeline configuration: 5 handlers in series, each performing ~100 µs of synthetic
  CPU work (tight loop accumulating 1000 additions — no I/O, no allocation)
- Lifecycle point tested: `pre_prompt` (iterates all handlers; short-circuits on first `Reject`)
- All handlers return `Allow`, so no short-circuit occurs and all 5 execute per call
- No-op metrics recorder (`NoOpMetricsRecorder`) — metrics overhead excluded by design

## Results — `hook.pre_prompt.5_handlers`

Criterion ran 100 samples across ~24M iterations (5.0 s collection window after 3 s warmup).

| Metric | Value |
|---|---|
| Mean latency (5 handlers) | 223.00 ns |
| 95% confidence interval | [217.41 ns, 229.47 ns] |
| P99 proxy (CI upper bound) | ~229.47 ns |
| Budget | 5,000,000 ns (5 ms) |
| Margin vs. budget | ~21,800x headroom |
| Outliers | 5 out of 100 samples (3 high mild, 2 high severe) |

The CI upper bound at 229.47 ns is used as a conservative P99 proxy. Criterion's default
output does not surface a P99 percentile directly; the upper confidence interval bound is
the most conservative single-call estimate available from the standard report.

## Verification

P99 proxy: 229.47 ns < 5,000,000 ns (5 ms) — **PASS**

The 5-handler pipeline comfortably satisfies the H09 latency invariant with more than
four orders of magnitude of headroom. The bottleneck at this call count is CPU-bound
handler work, not pipeline dispatch overhead.
