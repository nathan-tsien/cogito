## 1. 背景与目标

v0.2 Sprint 11 交付了同步 `delegate(role, input) -> output`:父 turn 在工具
invoke 内 inline await 子跑完,期间不能 spawn 兄弟、不能被干预。SaaS 形态下
父本身是 request handler,必须保持响应、能 fan-out、能让人取消失控的子 ——
v0.3 把父→子跳变成异步且可控,同时 v0.2 消费者的 `delegate` 行为逐字节不变。

**v0.3 交付物**:
1. 四个新工具:`spawn_agent` / `wait_agent` / `send_input` / `cancel_agent`
2. 父 log 事件树:`SubagentSpawned` / `SubagentInputSent` / `SubagentCompleted`
3. 父崩溃重启时对 outstanding 子的确定性对账
4. chaos 场景:父崩子续、子崩父收错

**设计总原则**:先定**终态交互模型**(§2),v0.3 实现它的严格前缀 ——
后续版本只增量、不推翻。这是针对"Codex v2 把 wait 默认语义翻转、
Claude Code stateless 措辞与 resume 矛盾"这两个业界返工教训的预防。

## 2. 终态交互模型(锚点,先于 v0.3 范围)

核心抽象:**父 spawn 出去之后,任何子的"可交付更新"到来时,父被唤醒一次、
处理一条。**

"更新"(update)是一个随版本增长的集合:

| 阶段 | 更新类型 | 父的唤醒方式 | 改动面 |
|---|---|---|---|
| **v0.3(本稿)** | 子终态(完成 / 失败 / 取消) | 父调 `wait_agent`(拉) | 本设计 |
| v0.4 | + 子中途消息(进度汇报、求助;子侧加上报工具,父侧 `SubagentMessageReceived` additive 事件) | 仍是 `wait_agent`(拉),内部 watcher 多 select 一路信箱 | 工具签名不变 |
| 未来可选 | 同上 | + 子更新直接触发父新 turn(推),骑 ADR-0016 TurnTrigger 抽象 | `wait_agent` 继续兼容存在 |
| 远期可选(teams 形态) | + 平级 agent 间消息 | 消息注入对话、收件方被唤醒跑 turn(同一底座原语) | 见下方"teams 形态的放宽点" |

因此 `wait_agent` 的语义从第一天就定义为
**"wait for the next update from any of the given children"**(多目标、
first-of、一次返回一条),而不是"等指定单个子结束"。后续演进只是更新类型
变多、触发方式变多,交互模式与工具签名不变。

对照:这与 Codex v2 `wait_agent`("Wait for a mailbox update from **any
live agent**, including queued messages and final-status notifications")
同构,与 Claude Code agent-teams 的消息驱动唤醒方向兼容;v0.3 子集 ≈
Codex v1 的能力面(调研报告 §4 对照表)。

**teams 形态的放宽点**(调研报告 §1.9):Claude Code agent teams 与层级
subagent 共享同一底座原语 —— "消息注入对话 + 收件方被唤醒跑一个 turn",
差异是三个放宽:拓扑树 → 图(子↔子直连,需通信授权模型)、生命周期
终态收口 → 常驻 idle(deregister 时机从"drive job 终态"变为"显式解散")、
协调父驱动 → 共享任务板自认领。本稿的设计在这三个放宽点上均不堵死:
子是可寻址一等 session(§3.5 共享 registry)、事件树按父子边记账(树→图
只是边变多)、`wait_agent` = 等下一条更新(更新来源扩到平级消息即可)。
teams 化是 v0.5+ 的产品决策,不在 v0.3 预埋机制;teams 的 token 经济性
教训(lead 被 idle 回声烧 13-22% input,claude-code issue #47930)反向
印证本稿 first-of 单条交付 + 读穿直返零成本的选择。

## 3. 方案比对

本节是评审重点:每个决策点列出候选、依据、结论。

### 3.1 交互模型:拉 vs 推 vs 隐式阻塞

| 候选 | 代表实现 | 机制 | 评估 |
|---|---|---|---|
| **A. 拉(显式 spawn/wait,模型编排)** | Codex v1 | 子产出经父模型显式 `wait_agent` 工具调用回流 context | 完全复用现有 async-tool 的 `TurnPaused -> JobCompletion -> TurnResumed`,**H01 零改动**;模型控制收割时机与顺序,context 顺序确定、KV-cache 友好 |
| B. 推(mailbox / 子完成自动重入父) | Codex v2、Claude Code 后台任务 | 子更新触发父新 turn 或注入 context | 需要"子更新"成为新的 turn 触发源(动 ADR-0016)、结果注入格式、父失去收割顺序控制;机制面大 |
| C. 隐式阻塞(spawn 即 wait) | Claude Code Agent 工具 | 一次工具调用 = spawn + 隐式等待,harness 批并发 | 即 v0.2 `delegate`,已有且保留;无法中途干预、无法乱序收割 |

**结论:v0.3 走 A,但 wait 语义按 §2 终态模型定义,使 B 成为 A 的增量
而非替代。** C 不动,作为 A 的糖继续存在。

### 3.2 wait 粒度:单目标 vs 多目标 first-of vs wait_all

| 候选 | 评估 |
|---|---|
| 单目标 `wait_agent(child)` | ADR-0011 草案原案。**否决**:fan-out 后逐个 wait,等 c1 时 c2 早完成了也拿不到 —— 收割顺序被迫等于 spawn 顺序,流水线性丢失;模型还要自己记账谁 outstanding |
| **多目标 first-of(`children` 缺省=全部)** | Codex v1 语义("multiple targets = first-to-finish")。先完成先交付,每轮唤醒处理一条真实负载;缺省全部 + 父 log 事件树记账,模型不需要记 outstanding 集合 |
| wait_all(收齐一起返回) | 省唤醒轮次,但最早完成的结果被最慢的子拖住,丢失流水线性;且"一次返回 N 条"的结果格式在 v0.4 加消息型更新后语义不稳 |

**结论:多目标 first-of,一次返回一条更新。** 模型循环调 `wait_agent`
直到收齐所需 —— 与 v0.4 mailbox 的逐条消费模式同构。

### 3.3 trait 形状:默认方法扩展 vs 新 trait vs 新 crate

| 候选 | 评估 |
|---|---|
| **A. `BrainSpawner` 加默认方法**(默认体返回 `SpawnError::Unsupported`) | v0.2 实现与测试替身零修改继续编译;`ExecCtx` 不加第二个 seam;**选定** |
| B. 独立 `SubagentLifecycle` trait | 编译期能力明确,但 `ExecCtx` 多一个 `Option<Arc<dyn ...>>`、双 trait 维护、"wired A 没 wired B"的半残状态更多 |
| C. 现在抽 `cogito-subagent` crate | 当前 301+114 LoC,v0.3 后仍 <1k;与 Runtime 内部(`open_inner`/`sessions`/`JobManager`)耦合紧,抽出要暴露 Runtime 内脏。**DEFERRED**(维持 ADR-0011 草案结论) |

### 3.4 multi-wait 机制:聚合 watcher job vs 协议扩展 vs FSM 扩展

multi-target wait 需要"等 N 个 job 中第一个",而现有 park 机制是单
`JobId`(`InvokeOutcome::Async(JobId)` -> `TurnPaused { job_id }`):

| 候选 | 评估 |
|---|---|
| **A. 聚合 watcher job** | `wait_agent` 内部 `submit_boxed` 一个 watcher future(select over N 个 `on_complete` sink),返回 watcher 自己的 job_id -> 单 job park 机制原样使用。协议、FSM、事件 schema 全不动 |
| B. `InvokeOutcome::AsyncAny(Vec<JobId>)` | 协议加变体、H01/H08/session loop 全要理解多 job park;事件 `TurnPaused` 也要变。面大,且 v0.4 watcher 加信箱路时它反而不够用 |
| C. H01 新增等待状态 | 违反"为一个工具加 FSM 状态"的克制;Inviolable #4 的状态集是设计资产 |

**结论:A。** watcher 是普通 job,v0.4 mailbox 化 = watcher 的 select
集合多一路,机制不变。

### 3.5 子句柄注册:共享 registry vs spawner 私有 map

| 候选 | 评估 |
|---|---|
| **A. 注册进 ADR-0034 共享 `sessions` DashMap** | 复用 v0.2.1 已落地的 `get_session`/`close_session` 生命周期;子对 surface 可寻址可观察(Codex 同路:子线程即普通线程,TUI 可切换观看);无新状态容器。**选定** |
| B. spawner 私有 per-parent map | 子对消费者不可见(封装好),但多一套生命周期管理代码,surface 无法观察、无法手动清理泄漏的子 |

代价与对策:A 使子对消费者可寻址,需文档约定"不要对子直接
`submit_user_text`,用 `send_input` 工具"(违者后果:绕过事件树,
`SubagentInputSent` 缺失;不会破坏正确性,只损失审计完整性)。

### 3.6 send_input 撞 mid-turn:排队 vs 报错 vs 排队+interrupt

| 候选 | 评估 |
|---|---|
| 仅排队(复用 single-slot) | session loop 已有 single-slot 语义(mid-turn 入槽、后来者覆盖、turn 末 drain),零新代码;**但**子正朝错误方向跑长 turn 时,纠正输入要等它跑完才生效 —— 太迟 |
| mid-turn 报错 | check-then-act 竞态;父模型被迫串行化 |
| **排队 + `interrupt` 可选参数** | 默认 false=排队(零新代码路径);`interrupt=true` -> `child.cancel_turn()`(`SessionHandle` v0.1 起已有)+ 输入入槽,子取消当前 turn 后立即消费新输入。Codex v1 同款("Use interrupt=true to redirect work immediately") |

**结论:排队为默认 + opt-in interrupt。** 终态子(无活 handle 或 log 已
终态)-> `ToolResult::Error`。ack 返回 `queued` / `started` / `interrupted`
三态。

### 3.7 父崩溃后 in-flight 子的处置:不自动重拔 vs opt-in 自动重拔

| 候选 | 评估 |
|---|---|
| **不自动重拔** | 写 `SubagentCompleted { outcome: Cancelled }`,parked `wait_agent` 返回结构化错误("subagent did not survive parent restart"),父模型决定是否重 spawn。与 v0.2 "模型决策错误处置"原则一致;非幂等子任务不会被盲重跑。**选定** |
| 策略级 opt-in 自动重拔(`respawn_on_crash`) | 多一个协议字段 + resume 路径分支;幂等性声明靠人保证,风险在框架侧。模型显式重 spawn 已覆盖该需求 |

### 3.8 子启动 context:fresh vs fork 父历史

| 候选 | 依据(调研报告 §4/§5) | 评估 |
|---|---|---|
| **fresh-context-only** | Claude Code 命名 subagent 永远 fresh;Codex v1 默认 fresh;Manus Wide Research "fresh, empty context window" | 双阀门语义不变,事件源清晰,KV-cache 友好。**选定** |
| fork 父历史(opt-in 或默认) | Codex v2 把默认翻成全量 fork 后,出现"子继续干父的活而不是委派任务"失效模式(issue #24150)+ kill-switch 诉求(#14981) | 反面教材。fork 留作未来 opt-in 增量,本版不做 |

### 3.9 背景设置机制(未决,挂 PR #72)

父模型如何为子打包背景 —— 候选机制(harness 注入环境事实 / role 级委派
提示 / `expected_output` 结构化输出约定 / 结构化 input 槽位 / 并发护栏),
业界依据见调研报告 §5,**待团队在 PR #72 收敛后以增量设计补入**。本稿
其余部分与该决策正交:任何一种背景机制都不动四工具骨架。

## 4. 工具面详细设计

`DelegateToolProvider` 从 1 个工具长到 5 个,同一 provider 的 `list()`
暴露、`invoke()` 按名路由。无新 crate、无 Brain 改动(H05 只是多见 4 个
`ToolDescriptor`)。所有工具共享深度护栏(§9)与
`ctx.brain_spawner.is_none()` 的"未接线"错误路径。

### 4.1 `spawn_agent`

```jsonc
// 入参
{ "role": "researcher", "input": "调研 X,需要…" }
// 返回(Sync)
{ "child_session_id": "…", "job_id": "…" }
```

- `ExecutionClass::AlwaysSync` —— 内部提交了后台 job,但对 H01 而言是同步
  返回 id 的工具,**不触发 TurnPaused**;模型可在同一 turn 连发多个 spawn。
- 流程见 §6.1。

### 4.2 `wait_agent`

```jsonc
// 入参(两个字段都可选)
{ "children": ["<id1>", "<id2>"],   // 缺省 = 全部 outstanding(已 spawn 且其终态尚未经 wait 交付)
  "timeout_secs": 600 }              // 缺省 = 无超时(子有各自的 drive 超时兜底)
// 返回(一条更新)
{ "child_session_id": "<id2>", "status": "completed", "output": "<子最终文本>" }
{ "child_session_id": "<id1>", "status": "failed",    "error": "…" }
{ "status": "timeout" }              // timeout_secs 到期且无更新
```

- 语义:**等 children 中下一条更新,first-of,一次一条**。模型循环调用
  直到收齐。参数对象可增字段(v0.4 `update_kinds` 等),additive。
- 双返回路径:
  - 任一目标子已终态且未交付过 -> **读穿直返**(`Sync`,不挂起,不 park);
  - 全部在跑 -> 聚合 watcher job(§6.2)-> `Async(watcher_job_id)` ->
    H01 照常 park。
- `children` 含未知 id(父 log 无对应 `SubagentSpawned`)-> `ToolResult::Error`。
- `children` 缺省且无 outstanding 子 -> `ToolResult::Error`("nothing to wait")。

### 4.3 `send_input`

```jsonc
// 入参
{ "child_session_id": "…", "input": "补充:…", "interrupt": false }
// 返回(Sync)
{ "ack": "queued" }       // 子 mid-turn,入 single-slot 槽(后来者覆盖,ack 提示)
{ "ack": "started" }      // 子空闲,立即触发新 turn
{ "ack": "interrupted" }  // interrupt=true:cancel_turn + 入槽
```

- 子已终态 / registry 中无活 handle -> `ToolResult::Error`。
- 排队覆盖前一条排队输入时,ack 附注 "replaced previously queued input"。
- 成功后父 log 写 `SubagentInputSent`;input 正文**不**入父 log(它作为子
  的 user turn 持久化在子 log —— 父 log 不做转录缓存,ADR-0007)。

### 4.4 `cancel_agent`

```jsonc
{ "child_session_id": "…" }  ->  { "ack": "cancelled" }
```

- `JobManager::cancel(drive_job)` + `child.shutdown(grace)` + deregister。
- **幂等**:取消已结束/已取消的子返回成功 ack,不是错误。
- 终态经 drive job 的 completion 路径写 `SubagentCompleted { Cancelled }`。

### 4.5 `delegate`(v0.2 契约不变)

- 同名、同 schema、同 `AlwaysSync`、同阻塞语义、同错误映射 —— v0.2 消费者
  字节级无感。
- 内部:`RuntimeSpawner::run_to_completion` 保留具体实现体;糖化为
  `spawn` + 单目标 inline wait 是后续内部重构,行为不变,非本版义务。

## 5. 协议变更(全部 additive,无 SCHEMA_VERSION bump)

### 5.1 `BrainSpawner` 加 4 个默认方法

```rust
#[async_trait::async_trait]
pub trait BrainSpawner: Send + Sync {
    // v0.2,不变。
    async fn run_to_completion(&self, req: DelegateRequest) -> Result<String, SpawnError>;

    // v0.3,additive;默认体保证 v0.2 实现零修改编译。
    /// Open the child, start its drive loop on a background job, register
    /// the child in the session registry, and arm the completion sink.
    async fn spawn(&self, req: DelegateRequest) -> Result<SpawnHandle, SpawnError> {
        Err(SpawnError::Unsupported { verb: "spawn" })
    }
    /// Resolve the drive job for an outstanding child (for wait parking).
    async fn drive_job(&self, child: SessionId) -> Result<JobId, SpawnError> { … }
    /// Push input into a live child. `interrupt` cancels its current turn first.
    async fn send_input(&self, child: SessionId, input: String, interrupt: bool)
        -> Result<SendInputAck, SpawnError> { … }
    /// Cancel the child's drive job and shut the child actor down.
    async fn cancel(&self, child: SessionId) -> Result<(), SpawnError> { … }
}
```

- 消费方:仅 `DelegateToolProvider` 的 5 个工具(经 `ExecCtx.brain_spawner`)。
  嵌入方(praxis 等)不直接调它 —— 它是"工具够到 Runtime"的协议桥。
- 实现方:`RuntimeSpawner(Arc<Runtime>)` 覆写全部;v0.2 测试替身
  (`MockSpawner`/`OkSpawner`)与任何消费者自定义 spawner 不改一行,
  新动词答 `Unsupported`。
- 对象安全:`#[async_trait]`、无泛型参数、`&self` 接收者 —— 与
  `Arc<dyn BrainSpawner>` 的既有用法兼容。
- 注:聚合 watcher(§6.2)由 `wait_agent` 工具组装,**不是** trait 方法 ——
  trait 保持最小动词集,组合逻辑留在工具层。

### 5.2 新协议类型

```rust
pub struct SpawnHandle { pub child_session_id: SessionId, pub job_id: JobId }
pub enum SendInputAck { Queued { replaced: bool }, Started, Interrupted }
// SpawnError(已 #[non_exhaustive])新增:
//   Unsupported { verb: &'static str }
//   UnknownChild { child: SessionId }
//   ChildTerminal { child: SessionId }
```

### 5.3 `EventPayload` 三个新变体(写入**父** log)

```rust
SubagentSpawned   { call_id: String, child_session_id: SessionId, job_id: JobId, role: String },
SubagentInputSent { child_session_id: SessionId },   // input 正文在子 log,不在此
SubagentCompleted { child_session_id: SessionId, outcome: JobOutcome },
```

沿 `JobSubmitted` / `ThinkingBlockRecorded`(ADR-0019)的 additive-variant
先例;`EventPayload` 是 `#[non_exhaustive]`,旧 log 读取不受影响。JSON
Schema 工件(`docs/schemas/conversation-event-v1.json`)再生成,CI drift
gate 重新钉住。

**事件序契约**:同一 `child_session_id` 在父 log 中恰好一对
`SubagentSpawned`/`SubagentCompleted`(后者可由 crash-resume 对账合成,
§7);`SubagentInputSent` 只能出现在两者之间。

## 6. 机制设计

### 6.1 spawn 流程

```
1. registry.get(role) -> 子 HarnessStrategy        (UnknownRole -> Error)
2. open_inner(child_id, New, strategy, child_meta, parent_depth+1,
              register = TRUE)                      <- v0.2 为 false;改为注册进 sessions DashMap(§3.5)
3. drive future(subscribe -> submit_user_text -> loop 至终态 -> shutdown)
   交给 job_mgr.submit_boxed(fut) -> job_id         <- v0.2 为 inline await
4. 立即 on_complete(job_id, 父 session actor 邮箱 sink)   <- 不等 wait_agent
5. 父 log 写 SubagentSpawned { call_id, child_session_id, job_id, role }
6. 工具返回 Sync({ child_session_id, job_id })
```

第 4 步是设计要点:**spawn 时就挂 completion sink**。子终态时父 session
loop 收到 `JobCompletionEvent`,据 job_id 匹配 outstanding 的
`SubagentSpawned`(replay 推导),写 `SubagentCompleted` 并从 registry
deregister 子 —— **事件树的完整性与句柄回收不依赖父模型是否调 wait**。
fire-and-forget 的子也被正确收口,不泄漏。

子 `SessionMeta` 沿 v0.2:`parent_session_id` / `parent_call_id` /
`subagent_depth = parent_depth + 1`,seq=0 持久化,replay 可重建。

### 6.2 聚合 watcher job(multi-wait 的核心)

```
wait_agent(children = [c1, c2, c3]):
  1. 逐个查:子 log 已终态 且 未交付过(§6.3)?
     是 -> 读穿:replay 提取最后 AssistantMessageAppended -> Sync 返回该条更新
  2. 全部在跑 -> 组装 watcher future:
       select over [c1, c2, c3 各自 drive job 的一次性 on_complete sink]
       第一个完成者 -> watcher 的 JobOutcome 携带 { child_id, outcome }
       (若给了 timeout_secs,select 多一路定时器 -> Timeout 型 outcome)
     job_mgr.submit_boxed(watcher) -> watcher_job_id
  3. 返回 Async(watcher_job_id) -> H01 单 job park,FSM/协议零改动
  4. JobCompletion -> TurnResumed -> 工具 resume 路径 replay 该 child 的
     log 提取产出,返回一条更新
```

- watcher 是普通 job:不进事件树(它不是子),不跨进程(crash 后见 §7)。
- 竞态安全:`on_complete` 契约是"到达终态即发、恰好一次"——若 select 挂上
  时某 job 已终态,sink 立即收到事件,无 lost-wakeup。
- 多个子几乎同时完成:watcher 只交付第一条;其余子的终态已由各自 spawn 时
  挂的 sink 写入 `SubagentCompleted`,下一轮 `wait_agent` 走读穿直返,
  零成本。

### 6.3 "已交付"集合的 log 推导(无 struct 状态)

`wait_agent` 缺省目标 = outstanding = 已 `SubagentSpawned` 减去
"其终态已经由某次 wait_agent 交付过"。交付记录天然落盘:每次 `wait_agent`
的返回作为 `ToolResultRecorded` 写入父 log,载荷含 `child_session_id`。
replay 时扫描 wait_agent 的工具结果即可重建已交付集合 —— 纯 log 推导,
符合 Inviolable #3,无任何跨 turn struct 状态。

### 6.4 深度与 ack 语义

- 深度:仅 `spawn_agent`(和 `delegate`)做 `ctx.subagent_depth >= max_depth`
  护栏(默认 3,`ToolsConfig.max_subagent_depth` 可配);
  `wait_agent`/`send_input`/`cancel_agent` 不加深树,无护栏。
- `send_input` 三态 ack 的判定:经 registry 拿活 handle;子 mid-turn ->
  入槽(覆盖时 `replaced: true`);空闲 -> 触发 turn;`interrupt=true` ->
  `cancel_turn()` + 入槽。

## 7. 父崩溃重启的对账(load-bearing)

异步 spawn 后,父可能带着 N 个 outstanding 子崩溃。恢复契约:

1. **父 log 是"哪些子存在"的唯一事实源。** replay 得
   `SubagentSpawned - SubagentCompleted` = outstanding 集合。这正是事件树
   非装饰性的原因:它是 Inviolable #3 要求的父侧锚点。
2. **`JobId` 不跨进程。** `JobManager` 在进程内;新 Runtime 的 job 注册表
   是空的 —— 旧 drive job 与旧 watcher job 一律视为**已丢失,不是 pending**。
   resume 不等它们。
3. **逐个 outstanding 子读其 log 对账**:
   - 子 log 有终态 turn -> 子其实跑完了(子 log 完好且权威):父侧补写
     `SubagentCompleted`(从子终态合成 outcome);后续 `wait_agent` 走
     读穿直返拿到产出。
   - 子 log 无终态 turn -> 子随旧进程死亡:补写
     `SubagentCompleted { outcome: Cancelled }`;后续 `wait_agent` 对它
     返回结构化错误("subagent did not survive parent restart"),父模型
     决定是否重 spawn(§3.7:不自动重拔)。
4. **parked 的父 turn 如何恢复**:父崩溃时若正 park 在 watcher job 上,
   resume 路径发现该 job 丢失 —— 对账完成后按现有 lost-job 恢复语义重入
   `wait_agent` 工具调用,此时全部目标子要么读穿可得、要么是合成的
   Cancelled 错误,**不会无限挂起**。
5. **无新 store 方法。** 以上全部是既有单 session `replay` 应用于父与各子
   id —— 跨 session 读由 backend 直接服务(Inviolable #7 / ADR-0007),
   JSONL 与 Postgres 同形。

净性质:崩溃的父带 N 个 outstanding 子,resume 是确定性的 —— 跑完的子从
其 log 收割,半路死的子标记 Cancelled 并以错误面呈模型;无跨进程 job
交接,父 log 中的子状态不超过 spawn/complete 两个标记。

## 8. registry 生命周期

- spawn 时注册(`open_inner(register = true)`),key = `child_session_id`。
- deregister 时机 = drive job 终态处理(完成/失败/取消皆同路);
  `cancel_agent` 与 `close_session` 幂等共存(ADR-0034 的关闭语义)。
- 消费者约定:子 session 对 `Runtime::get_session` 可见(可观察、可手动
  清理),但**不要**绕过工具直接 `submit_user_text` —— 违者损失
  `SubagentInputSent` 审计记录,不破坏正确性。

## 9. 护栏

- `max_subagent_depth`:默认 3,机制沿 v0.2(`ExecCtx.subagent_depth`
  来自 `SessionMeta`,replay 幸存)。
- 并发数护栏(Codex `max_threads=6` 同类):**挂 PR #72 一并讨论**;若做,
  实现为 `DelegateToolProvider` 构造参数 + replay 推导当前 outstanding 数,
  additive。
- 子 drive 超时:沿 v0.2 `CHILD_DRIVE_TIMEOUT = 300s` 兜底。**调优点**:
  异步子可能合法长跑,该值是否按 role/调用放宽(如 `spawn_agent` 入参或
  strategy 字段),留作实现期决定,默认值不变。

## 10. 测试策略

- **协议**:新类型/新事件变体 serde 回环;旧 log(无新变体)读取兼容。
- **contract**:`BrainSpawner` 默认方法对 v0.2 替身返回 `Unsupported`;
  `RuntimeSpawner` 全动词契约测试。
- **集成**(MockModelGateway):
  - fan-out:spawn x3 -> wait 循环收齐,乱序完成,断言 first-of 交付顺序
    = 完成顺序而非 spawn 顺序;
  - 读穿:wait 已完成子不产生 TurnPaused 事件;
  - send_input 三态 ack 各一例;interrupt 路径断言子当前 turn 被取消且新
    输入被消费;
  - cancel 幂等(双发、对已完成子发);
  - 事件序契约:Spawned/Completed 恰好一对,InputSent 居间;
  - fire-and-forget:spawn 后不 wait,断言 SubagentCompleted 仍落盘、
    registry 已 deregister。
- **resume chaos**(`make chaos` 新场景):
  - 父崩溃、子已完成 -> resume 合成 Completed、wait 读穿得产出;
  - 父崩溃、子半路死 -> resume 合成 Cancelled、wait 得结构化错误;
  - 父 park 在 watcher 上崩溃 -> resume 不挂起(§7.4);
  - 子崩溃(panic)-> 父收 `ToolResult::Error`,父 turn 继续。
- **回归**:v0.2 `delegate` 全部既有测试不动、全绿。
